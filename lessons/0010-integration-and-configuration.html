Lesson 10: Integration and Configuration

Objective: see how small tested modules compose into one application without losing testability.

Key ideas

  • The factory builds and wires real components from validated settings; the flows (run_enrollment, run_attendance) accept a components object and a frame source — so end-to-end tests inject fakes and never touch hardware.
  • Configuration is a boundary too: every tunable lives in one validated settings model, overridable via FA_* environment variables. Bad values fail at startup with the variable named, not mid-shift at a kiosk.
  • The CLI is a thin shell: parse arguments, build settings, dispatch, and map known errors to clear messages and exit codes. Business logic never lives in the CLI layer.
  • Operator experience is part of correctness: deduplicated status messages, a session summary (frames read/processed/dropped, events logged), and explicit warnings (empty gallery) beat silent behavior.

Retrieval question

Why does build_components not fail when model files are missing? (Answer: models load lazily on first frame, so commands that never touch the camera — report, employees list — keep working before models are downloaded; camera commands still fail loudly with a "run download_models" message.)
