Lesson 4: Camera Capture Loops

Objective: understand why camera code needs explicit lifecycle management and injectable backends.

Key ideas

  • A camera is an exclusive OS resource: if you do not release() it, the next process (or your next run) cannot open it. Context managers make cleanup automatic.
  • VideoCapture.read() failing is normal operation (device unplugged, driver hiccup) — treat it as an expected error path with a clear exception, not a crash.
  • Frame metadata (id, camera, timestamp, size) travels through the pipeline as a validated contract; the raw pixels stay in memory only and are never persisted.
  • Injecting the capture factory (capture_factory=cv2.VideoCapture by default) lets tests simulate every failure mode without hardware.

Retrieval question

Why does OpenCvCamera.read() raise instead of returning None when a frame fails? (Answer: silent None values propagate as mystery crashes later in the pipeline; failing loudly at the boundary points directly at the camera.)
