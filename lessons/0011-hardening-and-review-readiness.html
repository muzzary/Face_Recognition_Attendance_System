Lesson 11: Hardening and Review Readiness

Objective: learn what separates "it works on my machine" from "an engineer can trust, run, and extend this".

Key ideas

  • Documentation is a contract: the README states not just how to run the system but where it breaks — threshold trade-offs, the video-replay liveness gap, and the exact backpressure policy. Reviewers trust documented limits more than claimed perfection.
  • Every "magic number" needs a home and a rationale: 0.363 is OpenCV's published SFace operating point, liveness floors are IOD-normalized, and all of them are env-tunable without code edits.
  • Operational readiness: structured logs with context, session statistics, explicit exit codes, and a demo checklist that proves each rubric item reproducibly.
  • CI across the supported Python range (3.10 and 3.13) catches version drift the dev machine hides.

Retrieval question

Why document the video-replay weakness instead of staying quiet about it? (Answer: an evaluator or attacker will find it in minutes; documenting it shows you understand the threat model and turns a hidden flaw into an engineering decision with a stated mitigation path.)
