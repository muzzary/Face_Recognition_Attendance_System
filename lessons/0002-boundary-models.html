Boundary Models

Phase 2 teaches why data should be checked when it crosses from one module to another.

The Small Win

We create Pydantic models for the important attendance payloads: frames, detected faces, embeddings, employees, matches, liveness results, and attendance events.

Why Boundaries?

A camera module, detector, matcher, liveness checker, and storage layer all make promises to each other. A boundary model makes that promise executable: wrong shape, wrong type, impossible confidence score, or surprise raw image path gets rejected immediately.

Memory hook: validate at the doorway, not after bad data has walked through the whole building.

What Pydantic Gives Us

Pydantic models inherit from BaseModel, use type annotations as field definitions, and raise ValidationError when data cannot be parsed into the model. For this project, we also forbid extra fields so raw image paths cannot sneak into event payloads.
